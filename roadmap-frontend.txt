## Раздел 1. VS Code и расширения

### 📖 Теория (выжимка)

**VS Code** — это редактор кода. Сам по себе он умеет немного, но его суперсила в расширениях.

Что обязательно ставим и зачем:

| Расширение | Зачем нужно |
|------------|-------------|
| **Live Server** | Запускает локальный сервер. Сохранил файл — страница в браузере сама обновилась. Без этого пришлось бы вручную жать F5 |
| **Prettier** | Автоформатирование кода. Сделал «кашу» из отступов — сохранил — код стал красивым |
| **Auto Rename Tag** | Переименовал открывающий тег — закрывающий поменяется сам |
| **Color Highlight** | Подсвечивает цвета прямо в коде (видишь `#ff0000` — рядом красный квадратик) |

**Emmet** уже встроен в VS Code — это магия сокращений. Например, пишешь `ul>li*3` и жмёшь Tab → получаешь список из 3 пунктов. Очень ускоряет вёрстку.

**Важная настройка — Format on Save:** чтобы Prettier форматировал код автоматически при сохранении файла.

### 🛠️ Что делать пошагово

1. Скачай VS Code с официального сайта `code.visualstudio.com`, установи
2. Слева открой вкладку **Extensions** (иконка квадратиков) или `Ctrl+Shift+X`
3. Найди и установи по очереди: `Live Server`, `Prettier - Code formatter`, `Auto Rename Tag`, `Color Highlight`
4. Настрой Format on Save:
   - `Ctrl+,` (открыть Settings)
   - В поиске введи `format on save` → поставь галочку
   - В поиске введи `default formatter` → выбери `Prettier - Code formatter`

### ✅ Домашнее задание

1. Установи VS Code и все 4 расширения
2. Создай папку `learning` на диске, открой её в VS Code (`File → Open Folder`)
3. Создай файл `index.html`
4. Внутри напиши `!` и нажми Tab (это Emmet — развернётся базовый шаблон HTML)
5. Внутри `<body>` напиши `h1` + Tab, впиши «Привет, я учусь верстать»
6. Сохрани файл и **специально сломай отступы** (наставь лишних пробелов) → сохрани снова → убедись, что Prettier всё выровнял

## Раздел 2. Live Server и DevTools

### 📖 Теория (выжимка)

**Live Server** — твой локальный «браузерный движок» для разработки. Запускаешь страницу не двойным кликом по файлу, а через сервер. Это правильно и пригодится потом (некоторые вещи, например `fetch`, не работают при открытии файла напрямую).

**DevTools (инструменты разработчика)** — встроены в каждый браузер, открываются по `F12` или `Ctrl+Shift+I`. Это твой главный диагностический прибор. Самые важные вкладки:

| Вкладка | Для чего |
|---------|----------|
| **Elements** | Видишь HTML-структуру страницы и CSS каждого элемента. Можно менять стили «на лету» и сразу видеть результат |
| **Console** | Сюда выводятся ошибки JS и твои `console.log`. Первое место, куда смотришь, когда «что-то не работает» |
| **Network** | Видишь все запросы (загрузка картинок, отправка форм). Понадобится на этапе форм |
| **Device Toolbar** | Иконка телефона/планшета — эмуляция мобильных устройств для проверки адаптива |

**Главный навык:** научиться **инспектировать** элемент — клик правой кнопкой по любому элементу на странице → «Просмотреть код» (Inspect). Так ты изучаешь чужие сайты и отлаживаешь свои.

### 🛠️ Что делать пошагово

1. Открой свой `index.html` в VS Code
2. Правый клик по файлу в редакторе → **Open with Live Server** (или кнопка «Go Live» внизу справа)
3. Откроется браузер с твоей страницей. Обрати внимание на адрес: `127.0.0.1:5500` — это локальный сервер, а не путь к файлу
4. Не закрывая браузер, поменяй текст в `<h1>` в VS Code и сохрани → страница обновится сама
5. Нажми `F12` → открой вкладку **Elements**, найди свой `<h1>`
6. Дважды кликни по тексту прямо в DevTools и поменяй его → увидишь изменение на странице (но в файле оно не сохранится — это временно)

### ✅ Домашнее задание

1. Запусти свою страницу через Live Server
2. Проверь автообновление: меняй текст 2-3 раза, сохраняй, наблюдай
3. Зайди на любой свой любимый сайт, нажми `F12`, во вкладке Elements найди главный заголовок страницы и «поиграйся» — поменяй ему цвет через панель Styles справа
4. Включи **Device Toolbar** (иконка телефона) и посмотри, как сайт выглядит на мобильном
5. Открой вкладку **Console** и напиши там: `console.log('Привет, консоль')` → нажми Enter

## Раздел 3. Git и GitHub

### 📖 Теория (выжимка)

**Git** — система контроля версий. Простыми словами: она сохраняет «снимки» (коммиты) твоего проекта во времени. Можешь откатиться назад, видеть историю изменений, не бояться что-то сломать.

**GitHub** — облачный сервис, куда ты загружаешь свой код. Это и резервная копия, и твоё **портфолио** (работодатели/клиенты смотрят туда).

**Ключевые понятия:**
- **Репозиторий (repo)** — папка проекта под контролем Git
- **Commit** — сохранённый снимок изменений с описанием («что я сделал»)
- **Push** — отправка коммитов с компьютера на GitHub
- **Remote** — связь между локальной папкой и репозиторием на GitHub

**Базовый рабочий цикл (запомни наизусть):**
```bash
git add .                      # добавил все изменения в "корзину"
git commit -m "описание"       # сделал снимок
git push                       # отправил на GitHub
```

### 🛠️ Что делать пошагово

1. Скачай и установи Git с `git-scm.com`
2. Зарегистрируйся на `github.com`
3. Открой терминал в VS Code: `Ctrl + ~` (или `Terminal → New Terminal`)
4. Настрой Git (один раз навсегда):
```bash
git config --global user.name "Твоё Имя"
git config --global user.email "твой@email.com"
```
5. На GitHub нажми **New repository**, назови `learning`, создай (без галочек README пока)
6. В терминале VS Code, находясь в папке `learning`, выполни по порядку:
```bash
git init
git add .
git commit -m "Первый коммит: настройка окружения"
git branch -M main
git remote add origin https://github.com/ТВОЙ_ЛОГИН/learning.git
git push -u origin main
```
7. Обнови страницу репозитория на GitHub — там должны появиться твои файлы

> ⚠️ При первом push GitHub попросит авторизацию. Проще всего войти через всплывающее окно браузера, которое предложит VS Code.

### ✅ Домашнее задание

1. Установи Git, настрой имя и email
2. Создай репозиторий `learning` на GitHub
3. Залей туда свою папку с `index.html`
4. **Сделай тренировочный цикл:** поменяй текст в `index.html` → выполни `git add .` → `git commit -m "Изменил заголовок"` → `git push` → убедись, что на GitHub текст обновился
5. Открой на GitHub историю коммитов (вкладка с количеством commits) — посмотри свои 2 снимка

## Раздел 1. Семантическая структура HTML

### 📖 Теория (выжимка)

**Семантика** — это использование тегов по смыслу, а не просто `<div>` на всё подряд. Браузеры, поисковики и скринридеры понимают смысл страницы благодаря семантике. Это важно для **SEO** и доступности.

**Структурные теги:**
| Тег | Назначение |
|-----|------------|
| `<header>` | шапка сайта (логотип, меню) |
| `<nav>` | блок навигации |
| `<main>` | основное содержимое (одно на страницу!) |
| `<section>` | смысловой раздел страницы |
| `<article>` | самостоятельный блок (карточка, пост) |
| `<aside>` | боковой/дополнительный контент |
| `<footer>` | подвал сайта |

**Когда `<div>`, а когда `<section>`?**
- `<section>` — когда это **смысловой блок** (обычно с заголовком): «Услуги», «Отзывы», «О нас»
- `<div>` — когда это просто **обёртка для стилизации** без смысла (например, контейнер для центрирования)

**Типичная структура лендинга:**
```html
<header>...</header>
<main>
  <section class="hero">...</section>
  <section class="about">...</section>
  <section class="services">...</section>
  <section class="reviews">...</section>
</main>
<footer>...</footer>
```

### 🛠️ Что делать пошагово

1. Создай `index.html`, разверни шаблон через `!` + Tab
2. Собери «скелет» лендинга только из семантических тегов (без контента и стилей)
3. Внутри `<main>` сделай 4-5 секций
4. Открой в DevTools, посмотри на чистую структуру

### ✅ Домашнее задание

1. Свёрстай семантический каркас лендинга: шапка, hero-секция, 3-4 смысловых секции, футер
2. Внутри каждой секции добавь `<div class="container">` (это будущий центрирующий контейнер)
3. Объясни письменно для каждого блока, почему ты выбрал именно этот тег

## Раздел 2. Контент: текст, ссылки, изображения

### 📖 Теория (выжимка)

**Текстовые теги:**
- `<h1>`–`<h6>` — заголовки по иерархии. **`<h1>` один на страницу** (главный заголовок)
- `<p>` — абзац
- `<ul>` / `<ol>` / `<li>` — маркированный/нумерованный списки
- `<a href="">` — ссылка
- `<span>` — строчная обёртка (выделить часть текста для стилей)

**Ссылки:**
```html
<a href="https://site.com" target="_blank" rel="noopener">Внешняя</a>
<a href="#services">Якорь (скролл к секции)</a>
<a href="tel:+79991234567">Позвонить</a>
<a href="mailto:mail@mail.ru">Написать</a>
```
> `target="_blank"` — открыть в новой вкладке. С ним всегда ставь `rel="noopener"` (безопасность).

**Изображения:**
```html
<img src="img/photo.jpg" alt="Описание картинки">
```
> **`alt` обязателен!** Это и для SEO, и для доступности, и показывается если картинка не загрузилась.

**Адаптивные картинки (продвинуто):**
```html
<picture>
  <source srcset="img/photo.webp" type="image/webp">
  <img src="img/photo.jpg" alt="...">
</picture>
```
Браузер сам выберет современный формат (WebP) если поддерживает, иначе — запасной JPG.

## Раздел 3. Метатеги и подключение ресурсов

### 📖 Теория (выжимка)

**`<head>`** — невидимая часть страницы с настройками. Что там обязательно:

```html
<head>
  <meta charset="UTF-8">
  <meta name="viewport" content="width=device-width, initial-scale=1.0">
  <title>Заголовок вкладки</title>
  <meta name="description" content="Описание для поисковиков">
  <link rel="stylesheet" href="css/style.css">
</head>
```

| Тег | Зачем |
|-----|-------|
| `charset="UTF-8"` | кодировка (чтобы кириллица не «ломалась») |
| `viewport` | **критично для адаптива!** без него мобильная версия не работает |
| `title` | текст вкладки + важно для SEO |
| `description` | описание в выдаче поисковика |

**Open Graph** (превью при отправке ссылки в соцсети/мессенджеры):
```html
<meta property="og:title" content="Заголовок">
<meta property="og:description" content="Описание">
<meta property="og:image" content="https://site.com/preview.jpg">
```

**Подключение шрифтов** (Google Fonts):
- Заходишь на `fonts.google.com`, выбираешь шрифт → получаешь `<link>` в `<head>`
- Либо `@import` / `@font-face` в CSS (для локальных шрифтов — предпочтительнее для скорости)

**Подключение CSS и JS:**
- CSS — в `<head>` через `<link>`
- JS — перед `</body>` или с атрибутом `defer`

### 🛠️ Что делать пошагово

1. Пропиши полный корректный `<head>` для своего лендинга
2. Добавь Open Graph теги, проверь превью (можно через сервисы валидации OG)

4. Подключи внешний файл `style.css`

## Раздел 4. CSS: селекторы, каскад, единицы измерения

### 📖 Теория (выжимка)

**Селекторы:**
```css
.class { }          /* по классу — основной способ */
#id { }             /* по id — редко, для уникальных */
tag { }             /* по тегу */
.parent .child { }  /* вложенный */
.a, .b { }          /* группировка */
```
> На лендингах стилизуй **через классы**, а не теги/id. Это гибко и предсказуемо.

**Каскад и специфичность** — если на элемент действуют разные правила, побеждает более «специфичное» (id > class > tag) и то, что ниже в коде. Не злоупотребляй `!important` — это «костыль».

**Единицы измерения (очень важно для адаптива!):**
| Единица | Что это | Когда |
|---------|---------|-------|
| `px` | абсолютные пиксели | border, мелкие фикс. размеры |
| `rem` | относительно размера шрифта корня (16px) | **отступы, шрифты — основной выбор** |
| `em` | относительно шрифта родителя | редко, для вложенных |
| `%` | процент от родителя | ширины блоков |
| `vw` / `vh` | % от ширины/высоты экрана | hero на весь экран, адаптив |

> 💡 `rem` — твой главный инструмент. `1rem = 16px` по умолчанию. Используй для шрифтов и отступов — тогда сайт легко масштабируется.

**`box-sizing: border-box`** — обязательная настройка. Делает так, что `padding` и `border` входят в заданную ширину (иначе блоки «разъезжаются»):
```css
* {
  box-sizing: border-box;
}
```

### 🛠️ Что делать пошагово

1. В начале `style.css` пропиши базовый сброс: `margin: 0`, `box-sizing: border-box`
2. Постилизуй элементы через классы
3. Переведи отступы и шрифты на `rem`
4. Сделай hero-секцию высотой `100vh`

3. Сделай hero высотой во весь экран (`100vh`)
4. Поэкспериментируй: задай блоку ширину в `%`, потом в `vw` — почувствуй разницу

## Раздел 5. Flexbox (главный инструмент раскладки) 🔥

### 📖 Теория (выжимка)

**Flexbox** — система выравнивания элементов в одну линию (ряд или колонку). Используется на КАЖДОМ лендинге: меню, карточки, кнопки, центрирование.

Включается на **родителе**:
```css
.parent {
  display: flex;
}
```

**Свойства родителя:**
| Свойство | Что делает |
|----------|------------|
| `flex-direction` | направление: `row` (по умолч.) / `column` |
| `justify-content` | выравнивание по **главной** оси (`center`, `space-between`, `space-around`, `flex-start`, `flex-end`) |
| `align-items` | выравнивание по **поперечной** оси (`center`, `stretch`, `flex-start`) |
| `gap` | расстояние между элементами (заменяет margin!) |
| `flex-wrap` | перенос на новую строку (`wrap`) |

**Свойства дочерних:**
- `flex-grow` — может ли элемент растягиваться
- `flex-shrink` — может ли сжиматься
- `flex-basis` — базовый размер
- `flex: 1` — частый приём (равномерно занять место)

**Самый частый приём — центрирование:**
```css
.center {
  display: flex;
  justify-content: center;
  align-items: center;
}
```

## Раздел 6. CSS Grid (двумерные раскладки) 🔥

### 📖 Теория (выжимка)

**Grid** — система для **двумерных** раскладок (строки И колонки одновременно). Где Flexbox — это «линия», Grid — это «таблица/сетка».

> 💡 Правило выбора: **Flexbox** — для элементов в линию (меню, кнопки). **Grid** — для сложных сеток (галереи, карточки в несколько рядов, целые макеты страниц).

Включается на родителе:
```css
.grid {
  display: grid;
  grid-template-columns: 1fr 1fr 1fr; /* 3 равные колонки */
  gap: 20px;
}
```

**Ключевые свойства:**
| Свойство | Что делает |
|----------|------------|
| `grid-template-columns` | задаёт колонки |
| `grid-template-rows` | задаёт строки |
| `gap` | расстояние между ячейками |

**Единица `fr`** — «доля» свободного места. `1fr 1fr` = две равные колонки.

**`repeat()`** — сокращение:
```css
grid-template-columns: repeat(3, 1fr); /* то же, что 1fr 1fr 1fr */
```

**Адаптивная сетка без media queries (мощный приём!):**
```css
grid-template-columns: repeat(auto-fit, minmax(250px, 1fr));
```
Карточки сами переносятся и подстраиваются под ширину экрана. Незаменимо для галерей.

## Раздел 7. Позиционирование

### 📖 Теория (выжимка)

**`position`** управляет тем, как элемент располагается относительно других.

| Значение | Что делает |
|----------|------------|
| `static` | по умолчанию, обычный поток |
| `relative` | сдвиг относительно своего места + становится «точкой отсчёта» для absolute-детей |
| `absolute` | вырывается из потока, позиционируется относительно ближайшего `relative`-родителя |
| `fixed` | прилипает к окну браузера (не двигается при скролле) |
| `sticky` | «липкое» — обычное, пока не доскроллишь, потом прилипает |

**Главный паттерн (родитель relative → ребёнок absolute):**
```css
.card {
  position: relative; /* точка отсчёта */
}
.badge {
  position: absolute; /* позиционируется внутри карточки */
  top: 10px;
  right: 10px;
}
```
Так делают бейджи «-50%», иконки на карточках, кнопки поверх картинок.

**`fixed`** — для прилипающей шапки или кнопки «наверх»:
```css
.header {
  position: fixed;
  top: 0;
  width: 100%;
}
```

**`sticky`** — для меню, которое прилипает при скролле:
```css
.menu {
  position: sticky;
  top: 0;
}
```

**`z-index`** — управляет «слоями» (что выше, что ниже). Работает только с позиционированными элементами.

### 🛠️ Что делать пошагово

4. Поэкспериментируй с `z-index` (наложи элементы друг на друга)

## Раздел 8. Адаптивность и типографика 🔥

### 📖 Теория (выжимка)

**Mobile-first** — подход, при котором сначала верстаешь под мобильный, потом расширяешь под десктоп. Это современный стандарт (большинство трафика — мобильный).

**Media queries** — правила для разных размеров экрана:
```css
/* базовые стили — мобильные */
.title { font-size: 1.5rem; }

/* для экранов от 768px и шире */
@media (min-width: 768px) {
  .title { font-size: 2.5rem; }
}
```

**Типичные брейкпоинты:**
- `480px` — телефоны
- `768px` — планшеты
- `1024px` — небольшие ноутбуки
- `1280px+` — десктоп

**`clamp()` — суперинструмент адаптивной типографики** (запомни обязательно!):
```css
font-size: clamp(1.5rem, 4vw, 3rem);
/*           мин,    идеал,  макс  */
```
Шрифт плавно меняется между минимумом и максимумом в зависимости от ширины экрана — **без media queries**. Используй для заголовков, отступов, ширины контейнеров.

**Контейнер** — типичный приём ограничения ширины контента:
```css
.container {
  max-width: 1200px;
  margin: 0 auto; /* центрирование */
  padding: 0 1rem; /* отступы по бокам на мобильных */
}
```

### 🛠️ Что делать пошагово

## Раздел 1. Transition (переходы)

### 📖 Теория (выжимка)

`transition` — это **плавный переход** свойства из одного состояния в другое. Без него любое изменение (например, цвета при наведении) происходит мгновенно, «рывком». С ним — плавно.

**Где живёт переход:** прописывается на самом элементе (в обычном состоянии), а не в `:hover`. Это важно — иначе при уводе мыши анимации не будет.

```css
.button {
  background: blue;
  transition: background 0.3s ease;
}
.button:hover {
  background: red; /* плавно перейдёт за 0.3 секунды */
}
```

**4 параметра transition:**
| Параметр | Что задаёт | Пример |
|----------|------------|--------|
| property | какое свойство анимировать | `background`, `transform`, `all` |
| duration | длительность | `0.3s`, `300ms` |
| timing-function | характер движения | `ease`, `linear`, `ease-in-out` |
| delay | задержка перед стартом | `0.1s` |

```css
transition: background 0.3s ease 0s;
```

**timing-function (характер движения)** — это важно для «живости»:
- `linear` — равномерно (часто выглядит «механически»)
- `ease` — медленно-быстро-медленно (по умолчанию, естественно)
- `ease-out` — быстрый старт, плавное торможение (хорошо для появлений)
- `cubic-bezier(...)` — своя кривая (можно настроить на `cubic-bezier.com`)

> ⚠️ Анимировать можно НЕ все свойства. Хорошо анимируются `opacity` и `transform` (они «дешёвые» для браузера). Свойства вроде `width`, `height`, `margin` анимировать можно, но они тормозят страницу — старайся заменять их на `transform`.

### 🛠️ Что делать пошагово

## Раздел 2. Transform (трансформации)

### 📖 Теория (выжимка)

`transform` — изменяет элемент в пространстве: двигает, поворачивает, масштабирует, наклоняет. Это **главный инструмент анимаций** — он быстрый и не «дёргает» соседние элементы (в отличие от изменения `margin` или `top`).

**Основные функции:**
| Функция | Что делает | Пример |
|---------|------------|--------|
| `translate(x, y)` | сдвигает | `translate(20px, -10px)` |
| `translateX / translateY` | сдвиг по одной оси | `translateY(-5px)` |
| `scale()` | масштабирует | `scale(1.1)` — увеличить на 10% |
| `rotate()` | поворачивает | `rotate(45deg)` |
| `skew()` | наклоняет | `skew(10deg)` |

**Можно комбинировать** в одну строку (порядок важен!):
```css
transform: translateY(-5px) scale(1.05) rotate(2deg);
```

**transform-origin** — точка, относительно которой происходит трансформация (по умолчанию центр). Например, `transform-origin: left` — поворот будет от левого края.

**3D-трансформации (для продвинутых эффектов):**
- `perspective` — задаёт «глубину» сцены (чем меньше значение, тем сильнее эффект)
- `rotateX / rotateY` — поворот в 3D (эффект переворота карточки)
- `translateZ` — приближение/отдаление

```css
.card {
  perspective: 1000px;
}
.card:hover .inner {
  transform: rotateY(180deg); /* эффект переворота */
}
```

### 🛠️ Что делать пошагово

1. Возьми картинку или блок, при наведении делай `scale(1.1)` + `transition`
2. Сделай блок, который при наведении слегка поворачивается и поднимается

## Раздел 3. Keyframes и animation (ключевые кадры)

### 📖 Теория (выжимка)

`transition` хорош для перехода «из А в Б». Но если нужна **сложная анимация** (несколько этапов, циклы, автозапуск без наведения) — нужны `@keyframes` + `animation`.

**`@keyframes`** описывает этапы анимации:
```css
@keyframes bounce {
  0%   { transform: translateY(0); }
  50%  { transform: translateY(-20px); }
  100% { transform: translateY(0); }
}
```
Можно использовать `from` / `to` вместо `0%` / `100%` для простых случаев.

**`animation`** подключает эти кадры к элементу:
```css
.ball {
  animation: bounce 1s ease-in-out infinite;
}
```

**Параметры animation:**
| Параметр | Что задаёт |
|----------|------------|
| name | имя из @keyframes |
| duration | длительность (`1s`) |
| timing-function | характер (`ease`, `linear`) |
| delay | задержка старта |
| iteration-count | сколько раз (`1`, `3`, `infinite`) |
| direction | направление (`normal`, `alternate` — туда-обратно) |
| fill-mode | состояние до/после (`forwards` — остаться в конечном кадре) |

```css
animation: bounce 1s ease-in-out 0s infinite alternate;
```

**Типичные задачи для лендингов:**
- Пульсация кнопки (привлечь внимание)
- Анимация загрузки (спиннер через `rotate` + `infinite`)
- Плавное появление блока (`opacity` 0→1 + `translateY`)
- Бесконечная «бегущая строка» / маркер
- Подпрыгивающая стрелка «скролл вниз»

### 🛠️ Что делать пошагово

## Раздел 4. Hover-эффекты для UI-элементов

### 📖 Теория (выжимка)

Это применение всего предыдущего на практике. Hover-эффекты — то, что делает лендинг «живым» и дорогим на вид. Главное правило: **эффекты должны быть быстрыми (0.2–0.4s) и плавными**, иначе раздражают.

**Псевдоклассы состояний:**
- `:hover` — наведение мыши
- `:focus` — элемент в фокусе (важно для инпутов и доступности)
- `:active` — в момент нажатия

**Псевдоэлементы для эффектов** (`::before`, `::after`) — позволяют добавить «фоновый слой» под текст для красивых заливок, не добавляя лишних тегов в HTML. Это база крутых hover-эффектов на кнопках.

```css
.btn {
  position: relative;
  overflow: hidden;
}
.btn::before {
  content: '';
  position: absolute;
  inset: 0;
  background: red;
  transform: translateX(-100%);
  transition: transform 0.3s;
}
.btn:hover::before {
  transform: translateX(0); /* заливка «выезжает» слева */
}
```

**Типичные приёмы:**
- Заливка кнопки, выезжающая с одной стороны
- Появление подчёркивания у ссылки от центра к краям
- Подсветка/тень у карточки (`box-shadow`)
- Сдвиг иконки внутри кнопки (стрелка едет вправо)
- Затемнение картинки + появление текста поверх неё

### 🛠️ Что делать пошагово

1. Сделай кнопку с заливкой через `::before` + `overflow: hidden`
2. Сделай ссылку с подчёркиванием, растущим от центра (`::after`, `transform: scaleX`)
3. Сделай карточку, у которой при наведении появляется тень и она слегка поднимается
4. Сделай картинку, которая при наведении затемняется и показывает текст

## Раздел 5. Бургер-меню и will-change (оптимизация)

### 📖 Теория (выжимка)

**Анимированный бургер** — иконка из 3 полосок, которая превращается в крестик. Это must-have для мобильной версии лендинга. На этом этапе делаем **только CSS-анимацию самой иконки** (превращение полосок в крестик). Логику открытия меню добавим позже на JS (Этап 3).

Суть: 3 полоски (`<span>`), при добавлении класса `.active`:
- верхняя поворачивается и опускается
- средняя исчезает (`opacity: 0`)
- нижняя поворачивается и поднимается
→ получается крестик. Всё на `transform` + `transition`.

**`will-change`** — подсказка браузеру: «этот элемент скоро будет анимироваться, подготовь его». Это оптимизация — браузер заранее выделяет ресурсы, анимация идёт плавнее.

```css
.animated-element {
  will-change: transform, opacity;
}
```

> ⚠️ Не злоупотребляй `will-change` — ставь только на реально анимируемые элементы. Если повесить на всё подряд — наоборот замедлит страницу.

**Производительность анимаций (важно понять сейчас):**
- Анимируй **только** `transform` и `opacity` — они работают на видеокарте (GPU), плавно
- Избегай анимации `width`, `height`, `top`, `left`, `margin` — они заставляют браузер пересчитывать всю страницу (тормозит)

3. Пока переключай класс вручную в DevTools (на JS перейдём позже)
4. Добавь `transition` для плавности превращения
5. Повесь `will-change: transform` на полоски

## Раздел 1. Основы языка: переменные, типы, операторы

### 📖 Теория (выжимка)

**Переменные** — это «коробки» для хранения данных.
- `const` — значение нельзя переназначить (используй по умолчанию ВСЕГДА)
- `let` — значение можно менять (используй, когда реально нужно менять)
- `var` — устаревшее, **не используй**

```js
const name = "Иван";   // не изменится
let count = 0;         // будет меняться
count = 5;             // ок
```

**Типы данных (основные):**
| Тип | Пример | Что это |
|-----|--------|---------|
| String | `"привет"` | строка (текст) |
| Number | `42`, `3.14` | число |
| Boolean | `true` / `false` | да/нет |
| null | `null` | «пусто» намеренно |
| undefined | `undefined` | значение не задано |
| Object | `{ }` | объект (набор данных) |
| Array | `[ ]` | массив (список) |

**Операторы:**
- Арифметика: `+ - * / %`
- Сравнение: `===` (строгое равенство, ВСЕГДА используй его, а не `==`), `!==`, `> < >= <=`
- Логические: `&&` (и), `||` (или), `!` (не)

**Шаблонные строки** (очень удобно):
```js
const name = "Иван";
console.log(`Привет, ${name}!`); // Привет, Иван!
```

**`console.log()`** — твой главный инструмент отладки. Выводит что угодно в консоль.

## Раздел 2. Функции и управление потоком

### 📖 Теория (выжимка)

**Функция** — это «мини-программа», которую можно запускать многократно. Принимает данные (аргументы), возвращает результат.

**Обычная функция:**
```js
function sum(a, b) {
  return a + b;
}
sum(2, 3); // 5
```

**Стрелочная функция** (современный синтаксис, используется чаще всего):
```js
const sum = (a, b) => {
  return a + b;
};
// или коротко, если одна строка:
const sum = (a, b) => a + b;
```

**`return`** — возвращает результат из функции. Без него функция вернёт `undefined`.

**Условия (if / else):**
```js
if (age >= 18) {
  console.log("Взрослый");
} else {
  console.log("Несовершеннолетний");
}
```

**Тернарный оператор** (короткое условие):
```js
const status = age >= 18 ? "Взрослый" : "Ребёнок";
```

**Циклы** (повторение действий):
```js
for (let i = 0; i < 5; i++) {
  console.log(i); // 0,1,2,3,4
}
```

## Раздел 3. Массивы и объекты

### 📖 Теория (выжимка)

**Массив** — упорядоченный список значений. Индексация с **нуля**.
```js
const fruits = ["яблоко", "банан", "груша"];
fruits[0];        // "яблоко"
fruits.length;    // 3
fruits.push("слива"); // добавить в конец
```

**Полезные методы массивов (критично для лендингов!):**
| Метод | Что делает |
|-------|------------|
| `forEach()` | перебрать каждый элемент (выполнить действие) |
| `map()` | создать новый массив, преобразовав каждый элемент |
| `filter()` | создать новый массив только из подходящих элементов |
| `find()` | найти первый подходящий элемент |

```js
const numbers = [1, 2, 3, 4];
numbers.forEach(n => console.log(n));      // выведет каждое
const doubled = numbers.map(n => n * 2);   // [2,4,6,8]
const even = numbers.filter(n => n % 2 === 0); // [2,4]
```

**Объект** — набор пар «ключ: значение». Описывает «сущность».
```js
const user = {
  name: "Иван",
  age: 25,
  isAdmin: false
};
user.name;       // "Иван" (через точку)
user["age"];     // 25 (через скобки)
user.age = 26;   // изменить
```

В реальных лендингах данные форм, настройки слайдеров, ответы от сервера — всё это объекты и массивы. Это нужно понимать твёрдо.

5. Создай объект «пользователь», обратись к свойствам, измени одно

## Раздел 4. Работа с DOM (сердце лендинга) 🔥

### 📖 Теория (выжимка)

**DOM** — это представление HTML-страницы в виде объектов, которыми можно управлять через JS. Здесь начинается «настоящий» фронтенд.

**Поиск элементов:**
```js
document.querySelector(".class");      // первый подходящий
document.querySelector("#id");         // по id
document.querySelectorAll(".item");    // ВСЕ подходящие (список)
```
> Запомни: `querySelector` использует синтаксис CSS-селекторов. `.class`, `#id`, `tag` — как в CSS.

**Изменение содержимого и стилей:**
```js
const title = document.querySelector(".title");
title.textContent = "Новый текст";       // изменить текст
title.style.color = "red";                // изменить стиль (но лучше через классы!)
```

**Работа с классами (главный приём):**
```js
element.classList.add("active");     // добавить
element.classList.remove("active");  // убрать
element.classList.toggle("active");  // переключить (есть → убрать, нет → добавить)
element.classList.contains("active"); // проверить (true/false)
```
> 💡 Главный паттерн фронтенда: вся анимация/состояние описаны в CSS-классе, а JS просто **добавляет/убирает класс**. Помнишь бургер из Этапа 2? Вот так JS и будет вешать `.active`.

**Создание элементов:**
```js
const div = document.createElement("div");
div.textContent = "Привет";
document.body.append(div);
```

## Раздел 5. События (events) — интерактивность

### 📖 Теория (выжимка)

**Событие** — это действие пользователя (клик, скролл, ввод текста, отправка формы). JS «слушает» события и реагирует.

**`addEventListener`** — главный метод:
```js
const button = document.querySelector(".btn");
button.addEventListener("click", () => {
  console.log("Кнопка нажата!");
});
```

**Частые типы событий для лендингов:**
| Событие | Когда срабатывает |
|---------|-------------------|
| `click` | клик по элементу |
| `submit` | отправка формы |
| `input` | ввод в поле (на каждый символ) |
| `scroll` | прокрутка страницы |
| `mouseenter` / `mouseleave` | наведение/уход мыши |
| `keydown` | нажатие клавиши |

**Объект события (`event`)** — содержит инфу о произошедшем:
```js
button.addEventListener("click", (event) => {
  event.preventDefault(); // отменить стандартное поведение (важно для форм и ссылок!)
  console.log(event.target); // элемент, на котором сработало
});
```

**Делегирование событий** (важный приём для производительности): вешаешь один слушатель на родителя вместо множества на каждого ребёнка.
```js
list.addEventListener("click", (e) => {
  if (e.target.classList.contains("item")) {
    // обработали клик по любому .item
  }
});
```

### 🛠️ Что делать пошагово

1. Повесь `click` на кнопку → выведи сообщение в консоль
2. Сделай так, чтобы по клику элемент менял класс (`toggle`)

## Раздел 7. Асинхронность и fetch (подготовка к формам) 🔥

### 📖 Теория (выжимка)

Это мостик к Этапу 4 (формы). Чтобы отправлять данные на сервер/в Telegram/в CRM, нужно понимать асинхронность.

**Проблема:** запрос на сервер занимает время. JS не должен «зависать», ожидая ответ — он работает асинхронно (продолжает работу, а когда ответ придёт — обработает его).

**`fetch()`** — отправка HTTP-запросов:
```js
fetch("https://api.example.com/data")
  .then(response => response.json()) // распарсить ответ в JSON
  .then(data => console.log(data))   // работаем с данными
  .catch(error => console.log(error)); // ловим ошибку
```

**`async / await`** — современный, более читаемый способ (используй его):
```js
async function getData() {
  try {
    const response = await fetch("https://api.example.com/data");
    const data = await response.json();
    console.log(data);
  } catch (error) {
    console.log("Ошибка:", error);
  }
}
```

**POST-запрос (отправка данных — то, что нужно для форм):**
```js
const response = await fetch(url, {
  method: "POST",
  headers: { "Content-Type": "application/json" },
  body: JSON.stringify({ name: "Иван", phone: "+7..." })
});
```

**JSON** — формат обмена данными между фронтом и сервером. По сути, текст в виде объекта.
- `JSON.stringify(obj)` — объект → строку (для отправки)
- `JSON.parse(str)` — строку → объект (при получении)

> 💡 Не пугайся, если асинхронность кажется сложной. На этом этапе достаточно **понимать общий принцип** и уметь скопировать-адаптировать `fetch`. Глубокое понимание придёт на практике в Этапе 4.

### 🛠️ Что делать пошагово

2. Сделай GET-запрос через `fetch` + `.then`, выведи данные в консоль

## Раздел 1. HTML-формы: правильная структура

### 📖 Теория (выжимка)

**`<form>`** — контейнер формы. Внутри — поля ввода и кнопка отправки.

```html
<form action="#" method="POST">
```

**Типы input (важно использовать правильные!):**
| Тип | Для чего | Бонус |
|-----|----------|-------|
| `text` | имя, текст | — |
| `email` | почта | браузер сам проверяет формат |
| `tel` | телефон | на мобильном открывает цифровую клавиатуру |

| `password` | пароль | скрывает символы |
| `checkbox` | галочка (согласие) | — |

## Раздел 2. Встроенная валидация HTML5

### 📖 Теория (выжимка)

Браузер умеет проверять поля **сам**, без JS. Это первая линия защиты.

**Атрибуты валидации:**
| Атрибут | Что проверяет |
|---------|---------------|
| `required` | поле не пустое |
| `minlength` / `maxlength` | длина текста |
| `min` / `max` | диапазон чисел |
| `pattern` | соответствие регулярному выражению |
| `type="email"` | формат почты |

**Pattern для телефона (пример):**
```html
<input type="tel" pattern="\+?[0-9]{10,15}" required>
```

**Стилизация состояний через псевдоклассы:**
```css
input:invalid { border-color: red; }   /* невалидно */
input:valid { border-color: green; }   /* валидно */
input:focus { outline: 2px solid blue; } /* в фокусе */
```

> ⚠️ HTML5-валидация удобна, но её **недостаточно**. Пользователь может обойти её, сообщения некрасивые, на разных браузерах выглядят по-разному. Поэтому дальше делаем валидацию на JS. Но HTML5-атрибуты всё равно оставляй — как базовый слой.

## Раздел 3. JS-валидация и сбор данных

### 📖 Теория (выжимка)

Здесь соединяем знания из Этапа 3 (события, DOM) с формами. Это сердце рабочей формы.

**Перехват отправки формы:**
```js
const form = document.querySelector(".form");

form.addEventListener("submit", (e) => {
  e.preventDefault(); // ОБЯЗАТЕЛЬНО! отменяем перезагрузку страницы
  // дальше наша логика
});
```
> `e.preventDefault()` — без него страница перезагрузится и данные потеряются. Это критично.

**Сбор данных через FormData (современный способ):**
```js
const formData = new FormData(form);
const data = Object.fromEntries(formData); // превращаем в обычный объект
console.log(data); // { name: "Иван", phone: "+7...", ... }
```

**Или вручную по полям:**
```js
const name = form.querySelector("[name='name']").value.trim();
```
`.trim()` — убирает лишние пробелы по краям.

**Своя валидация:**
```js
const phone = form.phone.value.trim();
if (phone.length < 10) {
  showError("Введите корректный телефон");
  return; // прерываем отправку
}
```

**Регулярные выражения (regex)** — для проверки форматов:
```js
const emailRegex = /^[^\s@]+@[^\s@]+\.[^\s@]+$/;
if (!emailRegex.test(email)) { /* ошибка */ }
```
> Regex не нужно зубрить — бери готовые проверенные шаблоны для email/телефона.

**Показ ошибок пользователю** — добавляй/убирай классы (паттерн из Этапа 3!):
```js
input.classList.add("error");
errorText.textContent = "Заполните поле";
```

### 🛠️ Что делать пошагово

1. Перехвати `submit`, поставь `preventDefault`, выведи данные в консоль
2. Собери данные через `FormData` + `Object.fromEntries`
3. Напиши проверку: имя не пустое, телефон ≥ 10 цифр, email по regex
4. Если ошибка — подсвечивай поле и показывай текст ошибки
5. Если всё ок — выведи «Форма валидна» в консоль

## Раздел 4. Отправка на Email (самый частый сценарий) 🔥

### 📖 Теория (выжимка)

Чтобы отправить данные на почту, нужен **сервер-посредник** (с фронтенда напрямую письмо не отправить). Есть три рабочих пути:

**Вариант 1. Готовые сервисы (для старта — лучший выбор):**
- **Formspree**, **Getform**, **FormCarry** — регистрируешься, получаешь URL, форма шлёт данные туда, они приходят тебе на почту. Без бэкенда.

```js
const response = await fetch("https://formspree.io/f/ТВОЙ_ID", {
  method: "POST",
  headers: { "Content-Type": "application/json" },
  body: JSON.stringify(data)
});
```

**Вариант 2. PHP-обработчик** (если у клиента обычный хостинг):
- Маленький файл `mail.php` на сервере принимает данные и отправляет письмо функцией `mail()`. Классика для лендингов на хостингах вроде Beget, Timeweb.

**Вариант 3. EmailJS** — отправка писем прямо с фронтенда через их сервис (без своего сервера).

> 💡 Для большинства лендингов: **Formspree/EmailJS на старте**, **PHP-обработчик** — когда нужно полностью своё решение на хостинге клиента. Разберись минимум с двумя.

**Полный цикл отправки (запомни схему):**
```js
form.addEventListener("submit", async (e) => {
  e.preventDefault();
  // 1. валидация
  if (!valid) return;
  // 2. показать "Отправка..."
  // 3. fetch на сервис/сервер
  // 4. показать "Спасибо!" или "Ошибка"
  // 5. очистить форму
});
```

5. Попробуй EmailJS как альтернативу

## Раздел 5. Отправка в Telegram и CRM 🔥

### 📖 Теория (выжимка)

**Отправка в Telegram** — очень востребовано (клиент получает заявки мгновенно в телефон).

**Как работает (схема):**
1. Создаёшь бота через `@BotFather` → получаешь **токен**
2. Узнаёшь свой **chat_id** (через бота `@userinfobot`)
3. Отправляешь сообщение через Telegram Bot API:

```js
const TOKEN = "токен_бота";
const CHAT_ID = "твой_chat_id";
const text = `Новая заявка!\nИмя: ${data.name}\nТелефон: ${data.phone}`;

await fetch(`https://api.telegram.org/bot${TOKEN}/sendMessage`, {
  method: "POST",
  headers: { "Content-Type": "application/json" },
  body: JSON.stringify({ chat_id: CHAT_ID, text })
});
```

> ⚠️ **Безопасность:** токен бота нельзя «светить» в открытом фронтенд-коде на боевом сайте (его могут украсть). Для учёбы — можно, для продакшена — токен прячут на сервере-посреднике (PHP-прокси или серверлесс-функция). Запомни это на будущее.

**Отправка в CRM:**
- У большинства CRM (Битрикс24, amoCRM, Salebot) есть **вебхуки** — специальный URL, на который шлёшь POST-запрос с данными, и заявка создаётся в системе автоматически.
- Принцип тот же `fetch` POST. Разница только в URL и структуре данных (смотришь в документации конкретной CRM).

```js
await fetch(CRM_WEBHOOK_URL, {
  method: "POST",
  headers: { "Content-Type": "application/json" },
  body: JSON.stringify({ /* поля по формату CRM */ })
});
```

> 💡 Часто на практике делают **отправку сразу в несколько мест**: и на почту, и в Telegram. Используют `Promise.all([...])` для параллельной отправки.

## Раздел 6. UX формы: состояния и обратная связь

### 📖 Теория (выжимка)

Рабочая форма — это не только «данные ушли». Пользователь должен **понимать, что происходит**. Это то, что отличает любительскую форму от профессиональной.

**Состояния, которые нужно обработать:**
1. **Загрузка** — после клика кнопка блокируется и показывает «Отправка...» (чтобы не нажали дважды)
2. **Успех** — сообщение «Спасибо! Мы свяжемся с вами» + очистка формы
3. **Ошибка** — «Что-то пошло не так, попробуйте позже»

```js
button.disabled = true;
button.textContent = "Отправка...";

try {
  await fetch(...);
  showSuccess("Спасибо! Заявка отправлена");
  form.reset(); // очистить форму
} catch (error) {
  showError("Ошибка. Попробуйте позже");
} finally {
  button.disabled = false;
  button.textContent = "Отправить";
}
```

**`form.reset()`** — очищает все поля формы.

**Защита от спама:**
- **Honeypot** — скрытое поле, которое люди не видят, а боты заполняют. Если оно заполнено — это бот, не отправляем. Простой и эффективный приём.
- **reCAPTCHA** — для серьёзной защиты (Google).

**Хорошие практики UX:**
- Сообщение об успехе/ошибке — заметное, в попапе или прямо под формой
- Можно показать попап-«спасибо» (помнишь модалки из Этапа 3?)
- Маска для телефона (библиотека вроде IMask) — чтобы телефон вводился в формате `+7 (___) ___-__-__`

### 🛠️ Что делать пошагово

1. Добавь состояние «Отправка...» с блокировкой кнопки
2. Сделай сообщение об успехе + `form.reset()`
3. Сделай обработку ошибки
4. Добавь honeypot-поле (скрытое, проверка на бота)
5. Подключи маску телефона через библиотеку IMask
6. Свяжи с попапом-«спасибо» из Этапа 3

### ✅ Домашнее задание

1. Реализуй все 3 состояния формы (загрузка / успех / ошибка) с блокировкой кнопки
2. После успешной отправки — очисти форму и покажи попап «Спасибо за заявку»
3. Добавь honeypot для защиты от ботов

# 📅 ЭТАП 5. GSAP — продвинутые анимации

Вот ради чего многие и идут в верстку лендингов — «вау-анимации», за которые платят дороже. GSAP (GreenSock Animation Platform) — индустриальный стандарт. Помнишь, в Этапе 2 я говорил, что GSAP делает то же самое, но мощнее? Сейчас ты это увидишь. Разобьём на 7 разделов. По времени — 2-3 недели.

> 💡 Главное преимущество перед твоего понимания: всё, что ты делал в CSS-анимациях вручную (keyframes, timing, delay), в GSAP делается одной строкой кода — и при этом гораздо гибче, точнее и с возможностями, которых в CSS просто нет.

---

## Раздел 1. Подключение и базовые методы (tween)

### 📖 Теория (выжимка)

**Подключение** (проще всего через CDN):
```html
<script src="https://cdnjs.cloudflare.com/ajax/libs/gsap/3.12.5/gsap.min.js"></script>
```

**Tween** — это одна анимация (от англ. «in-between», промежуточные кадры). Три главных метода:

**`gsap.to()`** — анимировать К указанным значениям (из текущего состояния):
```js
gsap.to(".box", { x: 200, duration: 1 });
// сдвинуть .box на 200px вправо за 1 секунду
```

**`gsap.from()`** — анимировать ОТ указанных значений (к текущему). Идеально для появлений:
```js
gsap.from(".box", { opacity: 0, y: 50, duration: 1 });
// элемент появится: выедет снизу с прозрачности
```

**`gsap.fromTo()`** — задаёшь и начало, и конец явно:
```js
gsap.fromTo(".box", { opacity: 0 }, { opacity: 1, duration: 1 });
```

**Частые свойства для анимации:**
| Свойство GSAP | Что делает |
|---------------|------------|
| `x`, `y` | сдвиг (это `transform: translate`, но короче!) |
| `opacity` | прозрачность |
| `scale` | масштаб |
| `rotation` | поворот (в градусах) |
| `duration` | длительность (в секундах) |
| `delay` | задержка |
| `ease` | характер движения |

> 💡 GSAP сам использует `transform` под капотом — поэтому анимации производительные. `x: 200` = `transform: translateX(200px)`, но писать удобнее.

### 🛠️ Что делать пошагово

1. Подключи GSAP через CDN, проверь в консоли: набери `gsap` — должен быть объект
2. Сделай блок, анимируй его через `gsap.to()` (сдвиг + поворот)
3. Сделай появление блока через `gsap.from()` (opacity + y)
4. Попробуй `fromTo()` с явными начальным и конечным состояниями
5. Поиграй с `duration` и `delay`

## Раздел 2. Easing — характер движения

### 📖 Теория (выжимка)

**Ease** — то, что делает анимацию «живой» и дорогой. Это самая важная вещь для качества анимации. В CSS были `ease`, `cubic-bezier` — в GSAP их десятки готовых и они мощнее.

**Базовые типы:**
```js
gsap.to(".box", { x: 200, ease: "power2.out" });
```

| Ease | Характер |
|------|----------|
| `none` | равномерно (= linear) |
| `power1/2/3/4` | ускорение/замедление разной силы |
| `back` | «перелёт» назад и возврат (пружинистость) |
| `elastic` | резиновое колебание |
| `bounce` | отскок как мячик |
| `expo` | очень резкое |

**Суффиксы (важно!):**
- `.in` — медленный старт, разгон
- `.out` — быстрый старт, плавное торможение (**самый используемый для появлений**)
- `.inOut` — медленно по краям, быстро в середине

```js
ease: "back.out(1.7)"     // появление с пружинкой
ease: "elastic.out(1, 0.3)" // резиновый эффект
ease: "bounce.out"        // отскок
```

> 💡 Правило хорошего тона: для появлений элементов почти всегда `power2.out` или `power3.out`. Для игривых акцентов — `back.out`. Поиграй на **gsap.com/docs/v3/Eases** (визуализатор).

### 🛠️ Что делать пошагово

5. Поэкспериментируй с параметрами `elastic.out(1, 0.3)`

## Раздел 3. Timeline — управление последовательностями 🔥

### 📖 Теория (выжимка)

**Timeline** — самая мощная фишка GSAP. Это «дорожка времени», на которую нанизываешь анимации друг за другом. В CSS ты делал это вручную через `delay` для каждого элемента (помнишь, мучился с появлением блоков по очереди?). Timeline делает это элегантно.

**Создание timeline:**
```js
const tl = gsap.timeline();

tl.from(".title", { y: 50, opacity: 0, duration: 0.8 })
  .from(".subtitle", { y: 30, opacity: 0, duration: 0.6 })
  .from(".button", { scale: 0, duration: 0.5 });
```
Анимации выполнятся **по очереди** автоматически. Не нужно считать delay вручную!

**Position parameter** — точный контроль времени (третий аргумент):
```js
tl.from(".a", { x: 100 })
  .from(".b", { x: 100 }, "-=0.3")  // начать на 0.3с РАНЬШЕ конца предыдущей
  .from(".c", { x: 100 }, "<")      // начать ОДНОВРЕМЕННО с предыдущей
  .from(".d", { x: 100 }, "+=0.5"); // начать через 0.5с ПОСЛЕ
```

| Значение | Что значит |
|----------|------------|
| `"-=0.3"` | наложение (раньше на 0.3с) |
| `"+=0.5"` | пауза (позже на 0.5с) |
| `"<"` | одновременно с началом предыдущей |
| `">"` | в конце предыдущей |

**Управление timeline:**
```js
tl.play();    // запустить
tl.pause();   // пауза
tl.reverse(); // проиграть назад
tl.restart(); // с начала
```

**Настройки по умолчанию для всей timeline:**
```js
const tl = gsap.timeline({ defaults: { duration: 0.6, ease: "power2.out" } });
// теперь не нужно писать duration/ease в каждой анимации
```

### 🛠️ Что делать пошагово

1. Создай timeline, добавь 3-4 анимации подряд — посмотри последовательность
2. Поиграй с position parameter (`-=`, `+=`, `<`)
3. Задай `defaults` для всей timeline
4. Сделай кнопки play/pause/reverse для управления своей timeline
5. Сделай красивую анимацию появления hero (заголовок → подзаголовок → кнопка → картинка)

## Раздел 4. ScrollTrigger — анимации по скроллу 🔥🔥

### 📖 Теория (выжимка)

**ScrollTrigger** — это то, ради чего 90% людей учат GSAP. Анимации, срабатывающие при прокрутке страницы. Именно это создаёт «вау-эффект» современных лендингов.

**Подключение (отдельный плагин):**
```html
<script src="https://cdnjs.cloudflare.com/ajax/libs/gsap/3.12.5/ScrollTrigger.min.js"></script>
<script>gsap.registerPlugin(ScrollTrigger);</script>
```

**Базовое появление при скролле:**
```js
gsap.from(".card", {
  opacity: 0,
  y: 100,
  duration: 1,
  scrollTrigger: {
    trigger: ".card",      // за каким элементом следим
    start: "top 80%",      // когда верх элемента дойдёт до 80% экрана
  }
});
```

**Параметр `start: "top 80%"`** читается так: «когда **верх триггера** достигнет отметки **80% высоты вьюпорта**». Это момент запуска.

**Ключевые настройки ScrollTrigger:**
| Параметр | Что делает |
|----------|------------|
| `trigger` | элемент-триггер |
| `start` | когда начать (`"top 80%"`) |
| `end` | когда закончить (`"bottom 20%"`) |
| `toggleActions` | поведение при входе/выходе |
| `scrub` | привязка анимации к скроллу (см. ниже) |
| `markers: true` | показать маркеры для отладки (убери в продакшене!) |
| `pin` | «приколоть» элемент во время скролла |

**`scrub: true`** — магия: анимация **привязана к позиции скролла**. Крутишь вверх — анимация назад, вниз — вперёд. Создаёт эффект «управления мышкой».
```js
scrollTrigger: { trigger: ".section", start: "top center", end: "bottom center", scrub: true }
```

**`pin: true`** — закрепляет секцию на экране, пока внутри проигрывается анимация (эффект «застывшей» секции с движущимся контентом).

### 🛠️ Что делать пошагово

## Раздел 5. Stagger и анимация групп элементов

### 📖 Теория (выжимка)

**Stagger** — автоматическая задержка между анимацией элементов из группы. Вместо ручного `delay` для каждой карточки — одна строка.

```js
gsap.from(".card", {
  opacity: 0,
  y: 50,
  duration: 0.6,
  stagger: 0.15  // каждая следующая карточка стартует на 0.15с позже
});
```
Карточки появятся «волной» — одна за другой. Очень красивый и частый эффект.

**Продвинутый stagger (объект):**
```js
stagger: {
  amount: 1,        // общее время на всю волну распределить
  from: "center",   // откуда волна: "start"/"center"/"end"/"edges"/"random"
  grid: "auto"      // для сеток — волна по сетке
}
```

| `from` значение | Эффект |
|-----------------|--------|
| `"start"` | по очереди слева/сверху |
| `"center"` | от центра к краям |
| `"edges"` | от краёв к центру |
| `"random"` | в случайном порядке |

**Комбинация с ScrollTrigger** (самый частый реальный кейс):
```js
gsap.from(".card", {
  opacity: 0, y: 50, stagger: 0.15,
  scrollTrigger: { trigger: ".cards", start: "top 80%" }
});
```
Карточки появляются волной, когда доскроллишь до секции. Это «продаваемая» анимация.

### 🛠️ Что делать пошагово

1. Сделай 6 карточек, анимируй их с `stagger: 0.15` — увидишь волну
2. Поиграй с `from: "center"`, `"edges"`, `"random"`
3. Сделай сетку и попробуй `grid: "auto"` со stagger
4. Объедини stagger со ScrollTrigger — карточки появляются волной при доскролле

## Раздел 6. Интерактивные анимации и текстовые эффекты

### 📖 Теория (выжимка)

**Анимация по событиям** (соединяем с Этапом 3):
```js
const box = document.querySelector(".box");
box.addEventListener("mouseenter", () => {
  gsap.to(box, { scale: 1.1, duration: 0.3, ease: "power2.out" });
});
box.addEventListener("mouseleave", () => {
  gsap.to(box, { scale: 1, duration: 0.3 });
});
```

**`overwrite: "auto"`** — если анимации конфликтуют (быстро наводишь/убираешь), это предотвращает «дёрганье»:
```js
gsap.to(box, { scale: 1.1, overwrite: "auto" });
```

**Анимация текста по буквам/словам** — эффектный приём. Плагин **SplitText** (платный, но есть бесплатные аналоги вроде **SplitType**):
```js
const split = new SplitType(".title", { types: "chars" });
gsap.from(split.chars, { opacity: 0, y: 50, stagger: 0.03 });
// каждая буква появляется по очереди
```

**Анимация чисел (счётчик)** — частый кейс для блока «статистика»:
```js
gsap.to(".counter", {
  innerText: 1000,
  duration: 2,
  snap: { innerText: 1 }, // округлять до целых
  onUpdate: function() {
    // обновление текста
  }
});
```

**Параллакс мышью** — слой двигается за курсором:
```js
document.addEventListener("mousemove", (e) => {
  gsap.to(".layer", { x: e.clientX * 0.05, y: e.clientY * 0.05, duration: 0.5 });
});
```

### 🛠️ Что делать пошагово

2. Подключи SplitType, сделай появление заголовка по буквам

4. Сделай лёгкий параллакс-эффект на движение мыши

## Раздел 7. Производительность и сборка полной анимации лендинга

### 📖 Теория (выжимка)

Теперь собираем всё вместе в полноценный анимированный лендинг и учитываем производительность.

**Правила производительности GSAP:**
1. Анимируй `x`, `y`, `scale`, `opacity`, `rotation` — они на GPU (быстро). Избегай анимации `width`, `top`, `margin`
2. **`will-change`** ставит GSAP сам, вручную не нужно
3. Не плоди сотни ScrollTrigger без надобности
4. Используй `gsap.context()` для группировки и очистки (важно при работе в React, но полезно знать)

**Уважение к пользователю (`prefers-reduced-motion`)** — некоторым людям анимации вредят (вестибулярные нарушения). Хороший тон — отключать:
```js
const reduceMotion = window.matchMedia("(prefers-reduced-motion: reduce)").matches;
if (!reduceMotion) {
  // запускаем анимации
}
```

**Структура анимационного файла лендинга** (как организовать код):
```js
gsap.registerPlugin(ScrollTrigger);

// 1. Анимация загрузки (hero) — timeline
// 2. Анимации секций при скролле
// 3. Stagger для карточек
// 4. Интерактив (hover, счётчики)
// 5. Параллакс
```

**`ScrollTrigger.refresh()`** — вызывай, если контент/высота страницы меняется динамически (например, после загрузки изображений), чтобы триггеры пересчитались.

> ⚠️ **Мера — главное.** Перегруженный анимациями лендинг раздражает и тормозит. Профессионализм — не «сделать побольше движухи», а сделать **уместно и плавно**. 2-3 ярких акцента лучше, чем всё мельтешит.

### 🛠️ Что делать пошагово

1. Организуй весь анимационный код в один структурированный файл
2. Добавь проверку `prefers-reduced-motion`
3. Проверь производительность в DevTools (вкладка Performance) — анимации должны идти 60fps без рывков
4. Убери все `markers`
5. Протестируй на мобильном (анимации не должны тормозить)

## Раздел 1. Оптимизация изображений 🔥

### 📖 Теория (выжимка)

Изображения — это **80% веса** типичного лендинга. Главный фронт оптимизации.

**Современные форматы:**
| Формат | Когда |
|--------|-------|
| **WebP** | основной выбор сегодня — на 25-35% легче JPG при том же качестве |
| **AVIF** | ещё легче, но поддержка чуть хуже — как доп. вариант |
| JPG | фото (запасной для WebP) |
| PNG | только если нужна прозрачность без WebP |
| **SVG** | иконки, логотипы — векторные, весят копейки, не пикселят |

**Адаптивные изображения** (помнишь `<picture>` из Этапа 1?) — отдаём WebP с запасным JPG:
```html
<picture>
  <source srcset="img/hero.webp" type="image/webp">
  <img src="img/hero.jpg" alt="..." loading="lazy" width="800" height="600">
</picture>
```

**`srcset` — разные размеры под разные экраны** (не грузить огромную картинку на телефон):
```html
<img srcset="img/photo-400.jpg 400w, img/photo-800.jpg 800w"
     sizes="(max-width: 600px) 400px, 800px"
     src="img/photo-800.jpg" alt="...">
```

**Lazy loading** — картинки ниже экрана грузятся только при доскролле:
```html
<img src="..." loading="lazy" alt="...">
```
> Для hero-картинки (первый экран) `loading="lazy"` НЕ ставь — она нужна сразу.

**`width` и `height`** на картинках — важно! Резервируют место, предотвращают «прыжки» вёрстки при загрузке (метрика CLS).

**Инструменты сжатия:** Squoosh.app (онлайн, от Google), TinyPNG, конвертеры в WebP.

## Раздел 2. Оптимизация кода и загрузки

### 📖 Теория (выжимка)

**Подключение скриптов правильно:**
```html
<script src="script.js" defer></script>
```
| Атрибут | Что делает |
|---------|------------|
| `defer` | скрипт грузится параллельно, выполняется после загрузки HTML — **используй по умолчанию** |
| `async` | грузится параллельно, выполняется сразу как загрузится (для независимых скриптов, аналитики) |

**Критический CSS** — стили первого экрана можно вставить прямо в `<head>` (inline), чтобы первый экран отрисовался мгновенно. Остальное — отдельным файлом.

**Минификация** — удаление пробелов/переносов из CSS/JS для боевой версии (`style.min.css`). Делают сборщики или онлайн-минификаторы.

**Шрифты:**
- `font-display: swap` — текст показывается сразу системным шрифтом, потом подменяется (не «невидимый текст» при загрузке):
```css
@font-face {
  font-family: 'MyFont';
  src: url('font.woff2') format('woff2');
  font-display: swap;
}
```
- Формат **woff2** — самый лёгкий, основной
- `<link rel="preload">` для критичных шрифтов
- Не подключай 10 начертаний — только нужные

**Preconnect** к внешним доменам (Google Fonts, CDN) — ускоряет соединение:
```html
<link rel="preconnect" href="https://fonts.googleapis.com">
```

### 🛠️ Что делать пошагово

1. Подключи все скрипты с `defer`
2. Переведи шрифты на woff2 + `font-display: swap`
3. Добавь `preconnect` к внешним доменам
4. Убери неиспользуемые начертания шрифтов
5. Сделай минифицированные версии CSS/JS для боевой сборки
6. Попробуй вынести критический CSS в `<head>`

## Раздел 3. SEO-оптимизация 🔥

### 📖 Теория (выжимка)

**SEO** — чтобы сайт находился в Google/Яндекс. Для лендинга это критично (клиент хочет клиентов из поиска).

**Базовые теги (помнишь из Этапа 1, теперь системно):**
```html
<title>Студия ремонта в Москве — Ремонт под ключ за 30 дней</title>
<meta name="description" content="Делаем ремонт квартир под ключ...">
```
- `title` — 50-60 символов, главный ключевой запрос, уникальный
- `description` — 150-160 символов, цепляющее описание (влияет на кликабельность в выдаче)

**Семантика = SEO.** Правильные `<h1>` (один!), `<h2>`, структура из Этапа 1 — поисковики это любят.

**Заголовки по иерархии:** `<h1>` — главный (что за сайт), `<h2>` — секции, `<h3>` — подразделы. Не пропускай уровни, не используй для красоты.

**Атрибуты `alt`** у картинок — поисковики «читают» их (и это плюс к доступности).

**Open Graph** (из Этапа 4) — красивое превью при репосте в соцсети/мессенджеры.

**Структурированные данные (Schema.org / JSON-LD)** — подсказки поисковику о сути бизнеса (организация, услуги, отзывы, адрес). Дают «расширенные сниппеты» в выдаче:
```html
<script type="application/ld+json">
{ "@context": "https://schema.org", "@type": "Organization", "name": "...", "telephone": "..." }
</script>
```

**Технические файлы:**
- **`robots.txt`** — что можно/нельзя индексировать

## Раздел 4. Доступность (Accessibility, a11y)

### 📖 Теория (выжимка)

**Доступность** — чтобы сайтом могли пользоваться люди с ограничениями (незрячие со скринридерами, без мыши и т.д.). Это и этика, и плюс к SEO, и признак профессионализма.

**Основное:**
- **Семантика** (опять она!) — скринридеры ориентируются по тегам
- **`alt`** у картинок — озвучивается
- **`<label>`** у полей формы (из Этапа 4)
- **Контраст** текста и фона (не серый по светло-серому) — есть нормы WCAG
- **Навигация с клавиатуры** — по сайту можно пройти на Tab, фокус видимый
- **`:focus-visible`** — стиль фокуса для тех, кто на клавиатуре:
```css
a:focus-visible { outline: 2px solid blue; }
```

**ARIA-атрибуты** — для сложных элементов, где семантики не хватает:
```html
<button aria-label="Закрыть меню">✕</button>
<nav aria-label="Основная навигация">...</nav>
```
> `aria-label` даёт скринридеру понятное название там, где визуально только иконка.

**Что часто забывают:**
- `lang="ru"` на `<html>` (язык страницы)
- Не убирать `outline` совсем (оставлять видимый фокус)
- Кнопки делать через `<button>`, а не `<div onclick>`

### 🛠️ Что делать пошагово

1. Проверь `lang="ru"` на html
2. Пройди по сайту только на клавиатуре (Tab) — всё ли доступно, виден ли фокус
3. Проверь контраст текста (DevTools показывает)
4. Добавь `aria-label` иконочным кнопкам (бургер, закрытие модалки)
5. Убедись, что интерактив сделан на `<button>`/`<a>`, а не на `<div>`

### ✅ Домашнее задание

1. Проверь и обеспечь полную навигацию по лендингу с клавиатуры
2. Добавь видимый стиль фокуса (`:focus-visible`)
3. Проставь `aria-label` всем иконочным кнопкам
4. Проверь контрастность, исправь проблемные места
5. Убедись, что весь интерактив на семантичных тегах

## Раздел 5. Кроссбраузерность и адаптивное тестирование

### 📖 Теория (выжимка)

**Кроссбраузерность** — сайт должен одинаково работать в Chrome, Firefox, Safari, Edge и на разных устройствах. У клиента может быть iPhone с Safari, где что-то «поедет».

**Главные источники проблем:**
- **Safari** (особенно iOS) — самый капризный, свои баги с flex/grid, видео, `100vh`
- Старые версии браузеров
- Разные размеры экранов

**Автопрефиксы** — некоторым CSS-свойствам нужны вендорные префиксы (`-webkit-`, `-moz-`). Раньше писали руками, сейчас делает **Autoprefixer** (через сборщик или онлайн).

**Проблема `100vh` на мобильных** — на iOS адресная строка «съедает» высоту. Решение — новые единицы `dvh` (dynamic viewport height):
```css
height: 100vh;  /* запасной */
height: 100dvh; /* современный, учитывает интерфейс браузера */
```

**`@supports`** — проверка поддержки свойства браузером:
```css
@supports (display: grid) {
  .layout { display: grid; }
}
```

**Тестирование:**
- DevTools Device Toolbar (быстрая проверка размеров)
- Реальные устройства (телефон, планшет — обязательно!)
- BrowserStack / аналоги — для проверки в браузерах, которых нет под рукой

### 🛠️ Что делать пошагово

1. Открой лендинг в Chrome, Firefox, Safari (если есть Mac/iPhone) — сравни
2. Прогони CSS через Autoprefixer
3. Проверь hero-высоту на мобильном, при необходимости используй `dvh`
4. Протестируй на реальном телефоне
5. Проверь все интерактивы (меню, форма, модалки, анимации) в разных браузерах

### ✅ Домашнее задание

1. Протестируй лендинг минимум в 3 браузерах, выпиши найденные различия
2. Прогони стили через Autoprefixer
3. Исправь проблему с высотой на мобильных (`dvh`)
4. Протестируй на реальном мобильном устройстве, исправь всё, что «поехало»
5. Составь чек-лист «что проверять перед сдачей» для себя

## Раздел 6. Аудит производительности (Lighthouse) 🔥

### 📖 Теория (выжимка)

**Lighthouse** — встроенный в Chrome инструмент аудита (DevTools → вкладка Lighthouse). Даёт оценку по 4 категориям и **конкретные рекомендации**. Это твой главный экзаменатор.

**Категории:**
| Категория | Что оценивает |
|-----------|---------------|
| **Performance** | скорость загрузки |
| **Accessibility** | доступность |
| **Best Practices** | правильность реализации |
| **SEO** | поисковая оптимизация |

> Цель для хорошего лендинга — **90+ по всем категориям** (зелёная зона).

**Core Web Vitals** — ключевые метрики Google (влияют на ранжирование!):
| Метрика | Что значит | Норма |
|---------|------------|-------|
| **LCP** | время отрисовки главного контента | < 2.5с |
| **CLS** | «прыжки» вёрстки при загрузке | < 0.1 |
| **INP** | отзывчивость на действия | < 200мс |

**Как улучшать (всё из прошлых разделов!):**
- Низкий Performance → оптимизируй картинки, defer скриптов, шрифты
- Высокий CLS → проставь width/height картинкам, резервируй место
- Низкий SEO → title/description/alt/semantic
- Низкий Accessibility → контраст, alt, label, aria

> 💡 Lighthouse не просто ставит оценку — он **списком пишет, что именно чинить**. Это твоя пошаговая инструкция по доведению до идеала.

2. Запиши оценки по 4 категориям

4. Чини по списку, перепрогоняй после каждого блока правок
5. Добейся 90+ по всем категориям

## Раздел 7. Деплой на хостинг 🔥

### 📖 Теория (выжимка)

**Деплой** — публикация сайта в интернете, чтобы он был доступен по адресу. Финальный шаг, который делает работу «настоящей».

**Варианты хостинга:**

**1. Бесплатные (для портфолио, тестов):**
- **Netlify** / **Vercel** — перетащил папку или подключил GitHub → сайт онлайн за минуту, бесплатный поддомен, HTTPS автоматом. **Идеально для портфолио.**
- **GitHub Pages** — публикация прямо из репозитория

**2. Платные хостинги (для клиентов в РФ):**
- **Timeweb, Beget, Reg.ru** — обычный хостинг с поддержкой PHP (помнишь, для `mail.php` из Этапа 4 он нужен!)
- Заливка файлов через **FTP** (FileZilla) или панель управления

**Домен** — адрес сайта (`mysite.ru`). Покупается у регистратора (Reg.ru и др.), привязывается к хостингу.

**Что нужно знать про деплой:**
- **FTP/SFTP** — протокол загрузки файлов на хостинг
- **Структура файлов** на сервере (обычно папка `public_html` или `www`)
- **HTTPS/SSL** — сертификат для `https://` (на Netlify/Vercel автоматом, на хостингах — Let's Encrypt бесплатно)
- **Привязка домена** к хостингу (через DNS-записи)

> 💡 Стратегия: учишься и держишь портфолио на **Netlify/Vercel** (бесплатно, мгновенно). Клиентские проекты с PHP-формами — на **обычном хостинге** (Timeweb/Beget).

### 🛠️ Что делать пошагово

1. Зарегистрируйся на Netlify, задеплой лендинг перетаскиванием папки
2. Проверь, что сайт открывается по выданному адресу с HTTPS
3. Свяжи деплой с GitHub (push → автообновление сайта)
4. Изучи панель обычного хостинга (зарегистрируй пробный на Timeweb/Beget)
5. Залей сайт на хостинг через FTP (FileZilla) или файловый менеджер
6. Изучи, как привязывается домен (теоретически)

### ✅ Домашнее задание

1. Задеплой свой лендинг на Netlify или Vercel, получи рабочую ссылку
2. Настрой автодеплой через GitHub (изменил код → push → сайт обновился)
3. Зарегистрируй пробный обычный хостинг, залей туда копию через FTP
4. Проверь, что PHP-форма (`mail.php` из Этапа 4) работает на хостинге
5. Изучи процесс привязки домена, законспектируй шаги

## Раздел 1. Вёрстка с реальных макетов (Figma)

### 📖 Теория (выжимка)

99% коммерческих заказов приходят в виде **макета в Figma**. Дизайнер рисует — ты верстаешь. Нужно уметь «читать» макет.

**Что нужно уметь в Figma (режим просмотра, бесплатно):**
- Смотреть **размеры** элементов (выделил → видишь width/height в пикселях)
- Снимать **цвета** (HEX/RGB) с любого элемента
- Узнавать **шрифты**: семейство, размер, насыщенность, межстрочный интервал
- Смотреть **отступы** между элементами (выделил элемент → Alt → навёл на соседний)
- **Экспортировать** картинки и иконки (Export → PNG/SVG/WebP)
- Видеть **сетку и контейнеры** (макет обычно строится на сетке 1200/1440px)

**Перевод макета в код (алгоритм):**
1. Сначала изучи весь макет целиком — пойми структуру (какие секции)
2. Раздели на блоки → семантическая разметка (Этап 1)
3. Определи контейнер и сетку
4. Верстай сверху вниз, секция за секцией
5. Сначала десктоп ИЛИ mobile-first — потом адаптив
6. Точность: отступы, шрифты, цвета — как в макете (это называется «пиксель-перфект»)

**Важно про адаптив:** часто дизайнер рисует только десктоп и мобильный, а планшет — на твоё усмотрение. Промежуточные состояния продумываешь сам.

3. Потренируйся снимать цвета, шрифты, размеры, отступы
4. Научись экспортировать картинки и иконки в нужных форматах
5. Свёрстай один макет полностью, максимально точно (пиксель-перфект)

## Раздел 3. Портфолио

**Что должно быть в портфолио:**
- **3-5 сильных работ** (лучше меньше, но качественных, чем 20 слабых)
- **Живые ссылки** на задеплоенные сайты (не скриншоты!)
- Желательно **ссылка на код** (GitHub) — показывает культуру кода
- Разнообразие: разные ниши, с анимациями и без, разной сложности

**Где размещать портфолио:**
1. **Свой сайт-портфолио** — лучший вариант (сам себе кейс! заодно показывает навык)
2. **GitHub** — репозитории с кодом + ссылки на демо
3. **Behance / Dribbble** — для визуала
4. Профили на биржах и в сообществах

**Как оформлять каждый кейс:**
- Скриншот/превью
- Что за проект, для какой ниши
- Что реализовано (форма в Telegram, анимации, адаптив)
- Живая ссылка
- Срок выполнения

## Раздел 4. Где брать заказы и как общаться с клиентом

### 📖 Теория (выжимка)

**Источники заказов (от простого к сложному):**

1. **Биржи фриланса** — Kwork, FL.ru, Workspace, Upwork (для англоязычных). Быстрый старт, но конкуренция и низкие чеки вначале.
2. **Telegram-чаты и каналы** с заказами на верстку/веб-разработку
3. **Сообщества дизайнеров** — дизайнеры постоянно ищут верстальщиков под свои макеты (отличный канал!)
4. **Сарафан** — самый ценный источник. Доволен клиент → рекомендует. Появляется после первых работ.
5. **Холодные предложения** — находишь бизнесы с плохими/старыми сайтами, предлагаешь сделать лучше.
6. **Соцсети** — ведёшь профиль, показываешь работы, привлекаешь клиентов.

**Общение с клиентом (важнейший навык):**
- **Выясни задачу до цены.** Не «сколько стоит лендинг», а «что за бизнес, какая цель, сколько экранов, есть ли макет, нужна ли форма/анимации».
- **Бриф** — список вопросов клиенту (ниша, цель, референсы, дедлайн, бюджет, что предоставляет).
- **Фиксируй договорённости** письменно (что входит в работу, сроки, цена, число правок).
- **Предоплата** — бери частичную предоплату (обычно 50%). Защита от «кидка».
- **Говори на языке выгод**, а не технологий. Клиенту не нужен «GSAP» — ему нужно «современный сайт, который выделит вас среди конкурентов и принесёт заявки».

> ⚠️ Главная ошибка новичков — **не уточнить объём работ и согласиться на бесконечные правки**. Чётко фиксируй: «в стоимость входит N правок, далее — доплата».

## Раздел 5. Ценообразование и оформление работы

### 📖 Теория (выжимка)

**Как считать цену (подходы):**
- **За проект** (фикс) — самый частый для лендингов. «Лендинг под ключ — N рублей».
- **За час** — реже, для доработок/правок существующих сайтов.
- **Пакеты** — «Базовый / Стандарт / Премиум» с разным наполнением (хороший приём: клиент выбирает, а не торгуется).

**На что смотреть при оценке:**
- Количество экранов/секций
- Сложность анимаций
- Адаптив (сколько версий)
- Форма и интеграции (почта, Telegram, CRM)
- Делаешь ли с готового макета или нужен дизайн
- Срочность (срочно = дороже)
- Количество правок

**Динамика цен новичка:**
1. Первые 2-3 заказа — можно дёшево/почти даром (ради отзывов и портфолио). Это **инвестиция**, не работа за бесплатно навсегда.
2. Появились отзывы → поднимаешь цену.
3. Очередь из клиентов → поднимаешь ещё.

> 💡 Не демпингуй слишком долго. Низкая цена привлекает **самых проблемных клиентов**. Адекватная цена = адекватные заказчики.

**Оформление работы:**
- **Договор / оферта** — даже простой, защищает обе стороны (особенно для крупных заказов)
- **ТЗ** — что именно делаешь (защита от «а добавь ещё вот это бесплатно»)
- **Этапы оплаты** — предоплата 50% → остаток после сдачи
- **Самозанятость** — оформи статус самозанятого (в РФ): легально, низкий налог, можешь давать чеки клиентам (повышает доверие)

### 🛠️ Что делать пошагово

1. Изучи цены на лендинги на биржах и у конкурентов в твоём регионе
2. Сформируй свою линейку: 3 пакета с разным наполнением и ценой
3. Составь шаблон простого ТЗ/договора
4. Реши схему оплаты (предоплата/постоплата)
5. Изучи оформление самозанятости

### ✅ Домашнее задание

1. Сформируй прайс из 3 пакетов (Базовый/Стандарт/Премиум) с чётким наполнением
2. Составь шаблон ТЗ и простого договора/оферты
3. Изучи и при готовности оформи самозанятость
4. Пропиши свою стратегию роста цен (от первых заказов до целевой ставки)

## Раздел 6. Процесс работы над заказом от А до Я

### 📖 Теория (выжимка)

Чёткий процесс = довольный клиент = повторные заказы и рекомендации.

**Этапы работы над заказом:**
1. **Бриф** — выясняешь задачу, цель, референсы, объём
2. **Оценка** — называешь срок и цену, фиксируешь объём работ
3. **Договорённость + предоплата** — ТЗ, аванс
4. **Получение материалов** — макет, тексты, картинки, логотип, доступы (почта/Telegram для формы)
5. **Вёрстка** — собираешь проект
6. **Промежуточная демонстрация** — показываешь черновик на твоём тестовом домене (Netlify), собираешь правки
7. **Правки** — вносишь согласованное число
8. **Финальная сдача** — деплой на хостинг клиента, проверка по чек-листу
9. **Постоплата**
10. **Поддержка** — мелкие правки после сдачи (оговори условия — платно/бесплатно и сколько)

**Демонстрация на промежуточном этапе** — критично! Показывай работу не в конце, а по ходу. Так не получится «я представлял всё иначе» в самом конце.

**Работа с правками:**
- Собирай правки **списком**, а не по одной (эффективнее)
- Отделяй правки «в рамках ТЗ» от «новых хотелок» (последние — за доплату)
- Будь вежлив, но не позволяй сесть на шею

> 💡 **Коммуникация решает больше, чем код.** Клиент простит небольшую техническую шероховатость, но не простит игнор сообщений и срыв сроков без предупреждения. Отвечай, предупреждай, держи в курсе.

1. Распиши свой регламент работы над заказом (по шагам выше)

4. Сделай шаблон для сбора правок (форма/таблица)
5. Пропиши условия поддержки после сдачи

## Раздел 7. Развитие и дальнейший рост

### 📖 Теория (выжимка)

Рынок не стоит на месте. После выхода на первые заказы важно **не остановиться**.

**Направления роста:**

1. **Углубление в текущем** — сложные анимации (продвинутый GSAP, WebGL/Three.js для 3D-эффектов), сложные интерактивы. Premium-сегмент платит больше.

2. **CMS и движки** — клиенты часто хотят **сами редактировать** контент. Освой:
   - **Tilda** (для быстрых проектов, многие клиенты на ней) — можно зарабатывать на сложных кастомных блоках через Zero-block + код
   - **WordPress** — огромный рынок, верстка тем
   - **Headless CMS** — современный подход

3. **JavaScript глубже + фреймворки** — следующий большой шаг. **React / Vue** открывают рынок веб-приложений и более дорогих заказов. Твоя база из Этапа 3 — фундамент для этого.

4. **Сборщики и инструменты** — **Vite**, препроцессоры (**SASS/SCSS**), **Git** на продвинутом уровне, БЭМ-методология. Ускоряют и профессионализируют работу.

5. **Soft skills и бренд** — ведение соцсетей, личный бренд, повышение чека, работа с постоянными клиентами, делегирование (когда заказов больше, чем успеваешь).

**Стратегия обучения дальше:**
- **Учись на реальных заказах** — каждый проект подкидывает новую задачу, гугли и решай
- **Разбирай чужие крутые сайты** (awwwards, behance) — повторяй приёмы
- **Не распыляйся** — выбери 1 направление роста и углубись, потом следующее

> 💡 Ты прошёл путь от нуля до коммерческого верстальщика. Дальше рост — это **спираль**: берёшь заказ → встречаешь новую задачу → учишься → растёшь в цене → берёшь заказ сложнее. Главное — не переставать практиковать.

2. Освой SCSS и сборщик Vite — они пригодятся сразу
3. Изучи основы одной CMS (Tilda или WordPress) — расширит круг клиентов
4. Составь план развития на 3-6 месяцев вперёд
5. Заведи привычку разбирать 1 крутой сайт в неделю

### ✅ Домашнее задание

1. Выбери одно направление роста и составь по нему мини-план обучения

## 💡 Главные советы по этапу 7

1. **Не жди «полной готовности».** Идеального момента «вот теперь я готов брать заказы» не будет. Бери первый заказ, когда есть 2-3 работы в портфолио — учишься в бою.
2. **Первые заказы — ради отзывов и опыта, не ради денег.** Это инвестиция в репутацию. Но не застревай в этом режиме надолго.
3. **Коммуникация важнее кода.** Отвечай быстро, предупреждай о задержках, держи клиента в курсе. Это создаёт репутацию надёжного человека.
4. **Фиксируй всё письменно** — объём, сроки, число правок, цену. Устные договорённости = будущие конфликты.
5. **Качество одной работы важнее количества.** Один вылизанный кейс в портфолио продаёт лучше пяти средних.
6. **Сарафан — твоя главная цель.** Сделай так, чтобы первые клиенты были в восторге → они приведут следующих. Это самый дешёвый и надёжный поток заказов.
7. **Учись постоянно.** Веб меняется быстро. Тот, кто перестал учиться, через год-два теряет конкурентоспособность.
